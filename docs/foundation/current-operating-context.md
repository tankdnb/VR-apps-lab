# Current Operating Context

- Date: `2026-04-19`
- Purpose: capture the repository state that a fresh session needs in order to
  continue useful work without relying on private chat history.

## What this repository is now

`VR-apps-lab` is a public:

- `knowledge repository`
- `pattern library`
- `working lab`

for VR utilities, overlays, diagnostics, tracking helpers, runtime tools, and
experimental XR integrations.

## Current repository shape

The repo is now centered on two durable layers:

1. `Foundation`
   Positioning, roadmap, operating rules, and public framing.
2. `Research system`
   Discovery, intake, catalog, families, methods, waves, and reuse plans.

Small donor-worthy examples can still be added in the future, but the mainline
repository should not be treated as one app-first codebase.

## Important recent cleanup

The repository no longer keeps the earlier app-first branch, legacy runtime
spikes, or old repository-path assumptions in tracked mainline content.

A new session should assume:

- the repo is documentation-heavy by design
- `research integrity` matters more than build-style validation for docs work
- new GitHub work should follow the established `wave` process
- every new study should update registry, families, methods, and backlog

## Canonical naming and path assumptions

- Canonical project name: `VR-apps-lab`
- Canonical GitHub repository:
  [tankdnb/VR-apps-lab](https://github.com/tankdnb/VR-apps-lab)
- Preferred local path:

## What a new session should not assume

Do not assume:

- there is one hidden “main app” to finish
- the repository should be evaluated as one shipping product
- old runtime spikes or removed app-first artifacts are still canonical
- `docs` are secondary to a code tree

## Primary current directions

The strongest repository directions right now are:

- VR utility overlays and helper tools
- runtime and layer diagnostics
- tracking bridges and device tooling
- accessibility and communication surfaces
- structured public research on VR utility ecosystems
- reusable implementation methods and donor extraction
